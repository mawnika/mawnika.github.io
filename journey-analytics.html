the challenge

How can we help our clients make more informed decisions based on customer behavior?

This includes:
  • Surfacing and mapping customer journeys
  • Revealing customer needs, pain points, and goals
  • Supplementing customer journeys with a complete list of metrics
✨ my role ✨

I owned the design effort for this project from June to November 2022.

That includes user research, ideation, technical and business related meetings with cross-functional teams, creating wireframes and final mockups, and several rounds of usability testing with internal teams and external clients.

the results of our research: actionable use cases

The MVP needs to solve two main use cases: viewing and configuring the journey map.

📌 Use Case 1: I want to see the various journeys that customers take.

A customer journey is a mapping with each node representing a page that a customer clicks on during a session.
Before designing the journey map, I played around with different card designs for the page nodes. Page cards needed to include at least these data points:
  • page name
  • drop-off rate
  • any relevant metrics like conversion rate, purchase count, time spent on page etc.
With the page cards set up, I began to wrap my head around how a journey would look and laid out hypothetical customer paths in Figjam.

❗ An important challenge: Eliminating looping paths.

A path looping backwards on the sankey diagram is equivalent to a customer navigating to the previous page. For example, by clicking the browser 'Back' button.
Challenges in implementing looping flows:
  • Flowing "backwards" does not align with the underlying session data structure, which was in a “Page type -> Next page type” format.
  • Isolating and displaying the percentage of users that looped back to the same page would significantly increase the development effort.
  • They are visually and conceptually confusing (users never move backwards in time).
Solution
To eliminate the loops,
  1. I pulled them out as regular forward flows*, decluttering the journey map.
  2. and introduced interaction number labels to signify that each column represents the user's next step.
*A drawback of this is that the map now has multiple instances of the same page, which can increase the number of nodes by a lot, but we handle this later by introducing filters to narrow it down.
The result
The mapping is now truly linear, more accurately represents the underlying data and is less cluttered.
usability testing - round 1

I created clickable prototypes that the UX team used to conduct usability tests with 3 internal users.

We were curious about these aspects of the feature:
  1. Can the user navigate to the Journey Analytics feature?
  2. Is the user able to use the journey map?
  3. Can the user configure their journey to display key metrics?

Discovered pain points

  1. It was difficult to find relevant journeys for the customer behavior being observed. For example, if a user wanted to see how many customers purchased items, they would have to click on every page card to check if the next page was a Checkout page.
  2. It wasn’t clear that the Page Metrics were relevant to the page being clicked.
  3. The Legend being below the fold was hard to find and went unnoticed.
"I would've eventually figured it out on my own but I don't think it was evidently clear from [the] start how to use this page"
"In the software I'm a clicker, I explore things, I just wanna go kick the tires and see how things work even if it takes a few extra steps"
"Perhaps if there was a link from the Dashboard to Journey Analytics..."
"...a great visual tool!"

Our main takeaway from testing

We needed a way to let users to define, filter, and efficiently configure the mapping to view only the pages relevant to their goals.
This learning tied in nicely to our second defined use case which I discuss below.
iterating on the design

📌 Use Case 2: I want to view the optimal path based on a chosen KPI from a defined starting point.

Improvements based on our usability tests and the second use case:
  1. Added Start and End point filters and moved the configuration panel closer to the journey map since they are used in conjunction.
    Used a simpler sentence structure style for filtering.
  2. Moved the legend up so that it’s visible in the first screen fold.
  3. To convey that they are related, added a highlight to the Page Metrics card whenever the corresponding page card on the map is hovered over or clicked.

future scope

Future iterations and next steps

Reveal where customers came from
Displaying source channels like Google Search, Ads, Social Media etc. was omitted from the MVP since it would grow the journey map exponentially.
Add more filtration and search features to clean up the map
The size of the mapping increases with the number of webpages and so, navigating a journey map with over 20-30 pages could be made easier by including more granular filters and search features.
Get feedback from clients on MVP
The next steps would be to collect and iterate on feedback from interested clients (through more usability tests) and subsequently from our Beta users.
